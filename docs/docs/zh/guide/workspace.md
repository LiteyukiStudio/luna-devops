# 部署后下一步

平台跑起来以后，不需要马上配置所有外部系统。按这条顺序走，最不容易绕路。

## 1. 配置运行集群

运行集群是应用最终部署的位置。可以是 Kubernetes，也可以是轻量的 K3s。

第一次联调建议准备一个可测试的集群，并确认 kubeconfig 能从 API / worker 容器访问。

删除运行集群前，需要先把引用它的环境迁移到其他集群或删除；平台不会在删除集群时自动删除环境。

## 2. 配置镜像站

镜像站用于保存构建产物，也可以用于直接部署已有镜像。

如果只是体验部署流程，可以先使用已有镜像；如果要跑完整构建链路，再配置推送凭据。

## 3. 创建环境

环境通常对应 `dev`、`staging`、`prod`。创建项目空间时，平台会自动创建一个 `prod` 环境，默认 1 副本、0.5 核 CPU、0.5 GiB 内存；需要多阶段交付时再新增 `dev` 或 `staging` 环境。

环境绑定运行集群后，应用的部署配置就知道要发布到哪里。

删除环境前，需要先清理引用它的部署配置、访问入口和发布记录，避免运行链路留下不可维护的引用。

## 4. 创建部署配置

部署配置回答“这个应用怎么发布”：

- 用已有镜像，还是从仓库构建。
- 发布到哪个环境。
- 服务端口是多少。
- 是否需要环境变量、Secret 和配置文件。

## 5. 创建访问入口

访问入口把域名、路径、TLS 和后端服务连起来。创建后，在页面里查看状态，再用浏览器或 `curl` 验证。
