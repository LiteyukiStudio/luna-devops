# DevOps 平台项目技术栈要求

## 1. 文档目标

本文档统一描述 DevOps 平台项目本身的技术栈、工程组织和约束要求，作为后续初始化仓库、生成项目骨架、编写代码和制定 CI 检查的依据。

平台定位是一个前后端一体的 DevOps 控制平面，需要长期对接 GitHub、Gitea、DockerHub、Gitea Registry、Harbor、Casdoor、Kubernetes/K3s 和 Ingress。构建执行统一由平台 Builder 完成，不接入 GitHub Actions 或 Gitea Actions。

选型目标：

- 稳定优先：平台属于基础设施入口，依赖必须成熟、可维护、升级路径清晰。
- 集成优先：后端要方便对接 Kubernetes、代码库、制品库、认证系统和网关。
- 控制台优先：前端是高信息密度运维控制台，不是营销页面。
- MVP 优先：第一阶段先支撑单容器应用从代码到域名访问的完整闭环。
- 简洁优先：避免过早引入复杂框架、CRD、服务网格和重型流程引擎。

## 3. 后端要求

### 3.1 Go 后端

后端使用 Go。

原因：

- Kubernetes 生态天然偏 Go，官方客户端、资源类型、控制器生态都更成熟。
- 平台需要处理外部 API 调用、状态同步、集群资源查询和异步任务，Go 的并发模型适合这类工作。
- 单二进制部署简单，适合私有化、内网和轻量化场景。
- 工程约束强，长期维护成本低。

暂不选择 TypeScript/NestJS 作为后端主栈，因为后续 Kubernetes 深度集成、资源 watch、controller 或 operator 演进会更适合 Go。

### 3.2 HTTP 和 API

后端 HTTP 框架使用：

```text
Gin
```

API 风格：

```text
REST + OpenAPI
```

要求：

- 所有对前端开放的接口必须进入 OpenAPI 描述。
- 前端 API 类型和 client 从 OpenAPI 生成。
- API 资源围绕 Project、Application、Environment、Release、Registry、Cluster、GatewayRoute 等领域模型组织。
- 不在 MVP 阶段引入 GraphQL。

数据库访问使用：

```text
GORM
```

要求：

- 领域模型和数据库模型保持清晰边界。
- 复杂查询允许使用原生 SQL，但优先封装在 repository 层。
- 不在 handler 中直接散落数据库查询。
- 数据库迁移仍使用 `golang-migrate` 管理。

本地运行边界：

- `docker-compose-dev.yaml` 提供开发联调组件，使用独立 Compose 项目名 `liteyuki-devops-dev`，启动 PostgreSQL / Redis / worker / builder，并把 PostgreSQL / Redis 暴露到宿主机 `5432` / `6379`。
- `docker-compose-dev.yaml` 直接声明容器必需变量，不读取宿主机 `.env.development`；builder 不读取数据库配置，只保留 API/Redis/身份信息。容器专用连接地址直接写在 compose 的服务环境变量中，避免宿主机 `go run` 使用的 `localhost` 配置污染容器网络。
- `.env.example`、`.env`、`.env.development` 和 `.env.production` 必须用注释标明变量 scope：通用运行模式、宿主机运行时、Builder、构建网络策略、浏览器/前端公开入口，避免同名变量在宿主机和容器网络中被误用。
- `docker-compose.yaml` 提供完整容器化部署，API / worker / builder 通过容器网络访问 PostgreSQL / Redis，不将它们映射到宿主机端口；对外默认只暴露 web `8088`。
- 本地部署更新不得停止或替换开发环境进程；如部署端口冲突，优先调整部署端口或容器网络配置。
- 日常开发默认拓扑是：Vite dev server + Go API 在宿主机运行，PostgreSQL / Redis / worker / builder 由 `docker-compose-dev.yaml` 提供。
- Worker 用于联调部署、网关、DNS 检查和证书任务；涉及 Kubernetes/K3s 时使用本机 kubeconfig 或平台配置的运行集群。
- Builder 真实构建依赖 Docker executor / BuildKit executor 容器。MVP 阶段 Builder 通信仅支持 Redis stream：API 按标签、scope 和在线状态选择 Builder 后投递到专属 stream，Builder 消费任务并写事件，Worker 消费事件落库。
- 开发 compose 中的 builder 默认通过 Redis 连接平台任务队列；`BUILDER_AGENT_NAME` 是 Builder Agent 的唯一标识，同名 agent 会更新同一条 `builder_agents` 记录。`BUILDER_SCOPES` 控制可用范围，格式如 `global,project:proj1,user:uid1`；`BUILDER_LABELS` 控制能力标签，应用构建标签必须被 Builder 全部包含。容器内 builder 复用宿主机 Docker socket 时必须配置 `BUILDER_WORKSPACE_HOST_ROOT` 为宿主机绝对路径，避免 executor workspace 挂载到容器内路径。HTTP Builder 接入、平台签发 token 和外部 Builder 注册先不实现。
- 前端和 API 不要求放入开发 compose；它们改动最频繁，优先在宿主机运行，避免每次改前后端都重建完整容器栈。
- Compose 文件保持收敛：`docker-compose-dev.yaml` 用于开发联调，`docker-compose.yaml` 用于完整部署验收。新增 compose 文件必须有明确必要性并使用独立 `name`，避免多个场景共享默认项目名导致容器、volume 和端口状态混乱。

### 3.3 Kubernetes/K3s 集成

运行时集成使用：

```text
k8s.io/client-go
```

第一阶段操作标准 Kubernetes 资源：

- Namespace
- Deployment
- Service
- ConfigMap
- Secret
- Ingress
- Pod 状态
- Deployment rollout 状态

第一阶段不做 CRD/Operator。领域模型稳定后，再评估是否引入 `controller-runtime` 和 CRD。

多集群要求：

- 后端模型从第一版开始支持多个 Kubernetes/K3s 集群。
- 系统必须支持设置一个默认运行集群。
- MVP 产品体验默认按单集群使用。
- Environment 必须绑定 `runtimeClusterId`。
- 不允许在部署逻辑中写死全局 kubeconfig。
- 集群 credential 作为 write-only secret 提交，后端加密保存为内部引用，业务接口不回显引用字段。
- 每个集群可以有独立的默认 namespace、IngressClass 和域名后缀。

Project 与 Namespace 要求：

- 平台用户系统和 Kubernetes 资源解耦。
- User 不直接映射 Kubernetes Namespace。
- Project 是协作和资源归属边界。
- MVP 默认一个 Project 对应一个 Kubernetes Namespace。
- Namespace 默认命名为 `project-{projectSlug}`。
- Environment 保留 `namespace` 字段，MVP 默认继承 Project namespace。
- 平台层通过 ProjectMember 做权限判断。
- 平台使用自身 Kubernetes credential 操作资源，并记录审计。
- 所有平台托管的 Kubernetes 资源必须写入 project、app、env、managed-by labels。

### 3.4 数据库

数据库使用：

```text
PostgreSQL
```

数据库访问：

```text
GORM
```

迁移工具：

```text
golang-migrate
```

主要存储：

- 用户、团队和项目成员映射
- 项目和应用
- GitProvider、GitAccount 和代码仓库绑定
- CI pipeline 配置和运行记录
- 制品库配置和镜像版本
- 集群配置
- 默认集群配置
- 环境配置
- 网关路由
- 发布记录
- 审计记录
- 外部系统状态快照

Git 集成主数据：

- `GitProvider` 由平台管理员维护，描述 GitHub、Gitea、GitLab 等代码平台的类型、地址、授权方式和 OAuth/App 配置引用。
- GitHub/Gitea 的 OAuth 接入统一按 OAuth App / OAuth2 Application 引导，前端展示创建入口、Callback URL、Homepage URL 和 scopes。Gitea 这类自定义部署平台需要根据实例 URL 动态生成创建入口。
- `GitAccount` 归属平台用户，描述该用户连接的外部 Git 账号。token、refresh token 和仓库凭据只保存引用字段，不保存明文。
- `RepositoryBinding` 归属 Project/Application，明确绑定时使用的 GitProvider、GitAccount、owner、repo、cloneUrl 和默认分支，为后续 webhook 触发 BuildRun 打底。

公开站点配置：

- 使用轻量 KV 表存储无密级字符串配置。
- 后端维护允许配置的 key、默认值、描述、是否公开等定义，不允许前端任意写入未知配置项。
- 公开配置用于站点 title、logo、favicon、登录页副标题等元数据。
- 后端启动时读取公开配置并缓存在服务内存，后台保存配置后同步刷新缓存。
- 前端通过批量 keys 接口读取公开配置，返回 `Record<string, string>`。
- 公开配置不能存储密钥、Token、凭据或内部地址等敏感信息。

### 3.5 Redis 和异步任务

缓存和任务使用：

```text
Redis + Asynq
```

用途：

- 短期状态缓存
- 分布式锁
- Webhook 去重
- 异步任务队列
- 部署状态临时缓存

第一阶段任务类型：

- 处理 Git webhook 构建事件
- 调度平台 BuildRun
- 创建一次性 Kubernetes 构建 Job
- 同步构建 Job 状态和日志
- 同步 Harbor 镜像信息
- 等待 Kubernetes rollout
- 执行发布回滚
- 定时刷新集群资源状态

后续如果发布流程、审批流程和补偿逻辑明显变复杂，再评估 Temporal。

### 3.6 后端模块组织

后端第一阶段采用模块化单体 + 多进程部署。

```text
cmd/api
  - HTTP API
  - Webhook 接收
  - OAuth 回调
  - CRUD 管理
  - 权限校验
  - 任务投递

cmd/worker
  - 异步任务执行
  - 构建 Job 管理
  - 部署发布
  - 状态同步
  - 证书申请
  - 资源清理
```

约束：

- `api` 和 `worker` 使用同一套内部领域模块。
- `api` 与 `worker` 不通过内部 HTTP 互相调用。
- 跨进程协作通过 PostgreSQL 状态表和 Redis/Asynq 任务队列完成。
- 长耗时任务必须进入 worker，不在 HTTP 请求中同步执行。
- 构建/部署阶段的配置字段需要统一支持变量模板。后端使用统一变量渲染组件在最终执行阶段渲染真实值，禁止在 handler、worker、builder 中分散手写替换逻辑。
- 后续可以按队列拆出 `worker-build`、`worker-deploy`、`worker-sync`。
- 只有出现独立扩容、独立发布或强隔离需求时，才拆成独立微服务。

建议目录：

```text
cmd/
  api/
  worker/
internal/
  auth/
  project/
  application/
  repository/
  pipeline/
  registry/
  cluster/
  deployment/
  gateway/
  release/
  audit/
  config/
  secret/
  provider/
    casdoor/
    gitea/
    github/
    harbor/
    kubernetes/
migrations/
openapi/

web/
  src/
    app/
    api/
    auth/
    components/
    i18n/
    layouts/
    pages/
    styles/
```

要求：

- 仓库采用 monorepo。
- Go 后端在仓库根目录开发。
- 前端在 `web/` 目录开发。
- 业务服务依赖 provider 接口，不直接依赖具体外部系统 SDK。
- provider 只处理外部 API，不写业务规则。
- release/deployment 模块负责发布状态机。
- audit 模块统一记录敏感操作。
- secret 模块不得明文回显密钥。

### 3.7 Python 工具链

如果项目中出现 Python 脚本、工具或辅助服务，必须使用：

```text
uv
```

要求：

- 不直接使用 `pip install` 管理项目依赖。
- Python 项目必须包含 `pyproject.toml`。
- 锁文件使用 `uv.lock`。
- 执行脚本优先使用 `uv run`。
- 安装依赖使用 `uv add` 或 `uv sync`。
- Python 只作为工具或辅助服务时，也必须遵守该规则。

## 4. 前端要求

### 4.1 前端主栈

前端使用：

```text
Vite + React + TypeScript + Tailwind CSS + shadcn/ui
```

Node.js 版本建议：

```text
Node.js 24 LTS
```

要求：

- 使用 Vite SPA，不引入 Next.js 或 SSR。
- 使用 TypeScript，不使用 JavaScript 编写业务模块。
- 前端依赖安装、脚本执行必须使用 `pnpm`。

- ESLint 负责拦截共享模块的跨目录相对导入，保证规则可持续执行。

### 4.3 页面模块组织

前端模块按页面拆分。

示例：

```text
src/pages/applications/
  index.tsx
  route.tsx
  components/
    application-list-table.tsx
    application-create-dialog.tsx
    application-status-badge.tsx
  hooks/
    use-application-list.ts
    use-create-application.ts
  schemas/
    application-form-schema.ts
  locales/
    zh-CN.json
    en-US.json
```

约定：

- `index.tsx` 导出页面主组件。
- `route.tsx` 声明该页面路由配置。
- `components` 只放当前页面模块私有组件。
- `hooks` 只放当前页面模块私有 hooks。
- `schemas` 放表单和页面级校验 schema。
- `locales` 放页面模块私有文案。

组件上提规则：

- 只被一个页面使用的组件，留在该页面模块。
- 被两个及以上页面稳定复用后，移动到 `src/components/common`。
- shadcn/ui 生成的基础组件只放在 `src/components/ui`。
- 不允许在多个页面重复手写相同的表格空状态、错误提示、确认弹窗、页面标题、状态 Badge、详情块和表单布局。
- 重复出现的业务 UI 需要抽离为跨页面组件，例如 `PageHeader`、`EmptyState`、`ErrorState`、`ConfirmDialog`、`StatusBadge`、`ResourceSummary`。
- 状态展示必须使用有语义颜色的 `StatusValueBadge` 或带 `tone` 的 `StatusBadge`，覆盖集群健康、镜像站/外部连接健康、构建/部署/网关任务、Webhook/DNS/证书/镜像扫描、启用/禁用和校验状态；不得在列表、详情或卡片中直接显示纯文本状态。
- 抽离组件时优先保留清晰参数和组合能力，不要为了复用制造过度抽象。
- 页面模块负责组合业务流程，共享组件负责稳定 UI 片段，基础 UI 组件负责通用交互和样式。

列表组件规则：

- 列表行和展示卡片默认只负责展示、选择或进入详情，不承担隐式编辑入口。
- 编辑、删除、测试、绑定等操作必须使用明确按钮或菜单入口，避免误触和语义混乱。
- 涉及列表展示的页面默认采用分页查询，前后端都必须支持分页参数和分页响应。
- 后端列表接口支持 `page`、`pageSize`、`sortBy`、`sortOrder`，`sortOrder` 使用 `asc` / `desc`。
- 可搜索列表支持 `keyword` 或明确的字段级过滤参数。
- 响应使用统一分页结构，包含 `items`、`page`、`pageSize`、`total`。
- 服务端必须限制最大 `pageSize`，避免一次性拉取过大数据。
- 前端统一分页组件放在 `src/components/common`，至少支持上一页、下一页、页码、每页数量。
- 排序能力抽离为可复用参数模型，支持排序字段和正序/倒序。
- 搜索是否展示由页面业务决定，但接入方式保持一致。
- 批量选择能力按业务需要启用，不强制所有列表展示。
- 列表项视觉保持紧凑，摘要信息优先两行内完成；次要元信息优先用 Badge 聚合展示。

复用判断：

| 情况 | 处理方式 |
| --- | --- |
| 只在单个页面使用 | 保留在页面模块内 |
| 两个页面重复出现且结构稳定 | 抽离到 `src/components/common` |
| 三个及以上页面使用 | 必须抽离 |
| 属于设计系统基础能力 | 放到 `src/components/ui` |
| 只和某个领域强相关 | 放到对应领域或页面模块的 `components` |

### 4.4 API 和数据请求

API 使用 OpenAPI 生成类型和基础 client。

外部平台边界：

- 涉及 GitHub、Gitea、GitLab、Harbor、DockerHub、OIDC、Kubernetes、Traefik、AI Provider 等第三方/外部平台的读取、探测、搜索、状态同步和写操作，必须走后端 provider/service/API 适配、聚合或反代。
- 前端只调用平台后端 API，不允许在前端编排第三方平台 API，也不允许通过多个底层代理接口在前端拼装业务流程。
- 后端对外部平台调用统一做鉴权、权限校验、SSRF/egress 控制、脱敏错误和审计记录。

推荐组织：

```text
src/api/
  client.ts
  generated/
    schema.ts
    client.ts
```

请求规则：

- 查询类接口使用 TanStack Query 的 `useQuery`。
- 创建、更新、删除、发布、回滚等操作使用 `useMutation`。
- mutation 成功后通过 `queryClient.invalidateQueries` 刷新关联数据。
- 页面组件不直接拼接 fetch 请求。
- API 错误统一转换为前端错误对象，再交给页面或 toast 展示。

示例：

```ts
export function useApplicationList(projectId: string) {
  return useQuery({
    queryKey: ['applications', projectId],
    queryFn: () => api.applications.list({ projectId }),
  })
}
```

### 4.7 Toast 通知

通知使用：

```text
Sonner toast
```

适用场景：

- 操作成功：保存、创建、删除、发布、回滚。
- 操作失败：API 错误、权限不足、外部系统异常。
- 异步任务：构建已触发、部署已开始、回滚已提交。

不适用场景：

- 需要用户做决定的危险操作，使用 Dialog。
- 页面主体状态展示，使用页面内 Alert、Badge、Timeline 或 Table。
- 长日志输出，不使用 toast。

规则：

- 成功提示保持短句。
- 失败提示说明失败对象和主要原因。
- 危险操作必须先弹确认框，确认后再 toast 结果。
- toast 文案必须走 i18n。
- 普通自动关闭 toast 需要展示倒计时进度条，便于用户感知提示剩余展示时间。

示例：

```ts
toast.success(t('applications.toast.createSuccess'))
toast.error(t('applications.toast.createFailed'))
```

### 4.8 错误页面和错误组件

错误展示必须用户友好。

不允许：

- 直接展示后端原始错误对象。
- 直接展示 OIDC provider 的原始异常。
- 直接展示堆栈、SQL 错误、内部服务地址、secret key、token、trace 细节。
- 只显示“Error”“Unknown error”“Forbidden”这类没有下一步的文案。

必须统一处理的场景：

- OIDC 回调失败。
- OIDC `state` 不匹配。
- OIDC 授权码过期或已使用。
- 用户未命中 OIDC 允许组白名单。
- 用户邮箱域不在允许范围内。
- 用户未被邀请。
- 本地账号被禁用。
- 登录会话过期。
- 权限不足。
- 项目不存在或应用不存在。
- 后端服务不可用。
- 网络异常。

推荐组件：

```text
src/components/common/error-state.tsx
src/components/common/auth-error-page.tsx
src/components/common/forbidden-page.tsx
src/components/common/not-found-page.tsx
src/components/common/error-boundary.tsx
```

错误组件需要包含：

- 用户能理解的标题。
- 简短原因说明。
- 下一步操作。
- 返回入口，例如返回登录页、返回项目页、重新发起登录。
- 可选错误码或 request id，方便排查。

OIDC 错误示例：

```text
标题：无法完成登录
原因：你的账号不在允许访问本平台的组织或用户组中。
操作：请联系平台管理员添加 OIDC 组白名单，或使用已授权账号登录。
```

权限错误示例：

```text
标题：没有访问权限
原因：你当前不是该项目成员，或没有执行此操作的权限。
操作：返回项目列表，或联系项目管理员授予权限。
```

错误分层：

| 层级 | 处理方式 |
| --- | --- |
| 表单字段错误 | 表单内展示 |
| 普通操作失败 | toast + 保持当前页面 |
| 页面级加载失败 | `ErrorState` |
| 认证或准入失败 | `AuthErrorPage` |
| 权限不足 | `ForbiddenPage` |
| 路由不存在 | `NotFoundPage` |
| React 渲染异常 | `ErrorBoundary` |

所有用户可见错误文案必须走 i18n。

危险操作：

- 删除项目、删除应用、禁用账号等危险操作必须使用可复用 `ConfirmDialog` 二次确认。
- 确认完成后再执行 API 调用，并通过 toast 展示结果。

### 4.10 主题

主题从 MVP 第一版开始实现。

必须支持三态：

| 模式 | 说明 |
| --- | --- |
| `light` | 强制亮色主题 |
| `dark` | 强制暗色主题 |
| `system` | 跟随操作系统主题 |

实现要求：

- 使用 CSS Variables 承载颜色 token，兼容 shadcn/ui。
- 主题偏好写入 `localStorage`，刷新后保持用户选择。
- 当用户选择 `system` 时，页面加载时读取 `prefers-color-scheme`。
- 当用户选择 `system` 时，运行期间监听系统主题变化并实时切换。
- 默认值为 `system`。
- 在 HTML 根节点设置 `data-theme="light"` 或 `data-theme="dark"`。
- 额外在应用状态中保留用户选择的 `themeMode`: `light` / `dark` / `system`。
- 状态色保持语义固定，不随主色调变化。
- 桌面端用户信息、退出入口和主题切换控件放在侧边栏底部，避免顶部 header 承载过多全局控制。
- 主题切换优先使用三段式控件展示 light、system、dark 三态。
- 前端动效库使用 `motion`，参考 `neo-blog` 的短时长自然动效风格。
- 动效覆盖页面切换、列表项进入、弹窗打开关闭、主题切换滑块、按钮按压反馈。
- 桌面侧边栏使用二级分组导航，按 DevOps、个人工作区、系统管理等栏目组织，栏目之间用横线分隔。
- MVP 阶段侧边栏保持固定宽度，不默认提供折叠入口。
- 默认动效时长控制在 150ms 到 240ms，避免使用长时间或过度装饰性动画。
- 全局必须支持 `prefers-reduced-motion` 降级，减少动态效果时禁用非必要动画。

建议伪代码：

```ts
type ThemeMode = 'light' | 'dark' | 'system'
type ResolvedTheme = 'light' | 'dark'

function resolveTheme(mode: ThemeMode): ResolvedTheme {
  if (mode !== 'system')
    return mode

  return window.matchMedia('(prefers-color-scheme: dark)').matches
    ? 'dark'
    : 'light'
}
```

ThemeProvider 职责：

- 初始化 `themeMode`。
- 解析 `resolvedTheme`。
- 写入 `document.documentElement.dataset.theme`。
- 监听 `matchMedia('(prefers-color-scheme: dark)')` 的 change 事件。
- 在用户切换主题时更新 localStorage 和 DOM。

主色调扩展：

- MVP 内置固定主色为 Kubernetes 风格蓝：亮色主题 `#326CE5`，暗色主题 `#60A5FA`。
- 后续支持用户选择任意主色。
- 主色通过 HSL 推导 `primary`、`ring`、`accent`、`sidebar-active` 等变量。
- `success`、`warning`、`danger`、`running`、`failed` 等状态色不跟随主色变化。

`cn()` 工具放在：

```text
src/lib/cn.ts
```

建议实现：

```ts
import { clsx, type ClassValue } from 'clsx'
import { twMerge } from 'tailwind-merge'

export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs))
}
```

### 4.12 表单

表单使用：

```text
React Hook Form + Zod
```

规则：

- schema 放在页面模块的 `schemas` 目录。
- 表单默认值集中定义，避免散落在 JSX 中。
- 所有业务表单使用统一 `Field` 组件，必填项使用主题色 `*`，不使用红色星号。
- 不易理解的字段必须在 label 旁提供问询图标，hover 时展示简短说明；说明文案优先回答“填了会怎样、不填会怎样、多个值如何分隔、后续哪里使用”。
- 字段错误在 label 行右侧展示，默认隐藏，hover 或 focus 时显示；未满足校验前提交按钮使用 Tailwind disabled 样式降低强调。
- 表单使用 `mode: 'onChange'`，让按钮禁用状态和错误状态及时同步。
- 前端校验负责体验，后端校验负责可信边界。
- 密钥类字段可以从前端填写，但必须作为 write-only 输入处理；后端响应只返回 `xxxSet` 状态或安全引用，不得再次返回真实密钥。
- OIDC `Client Secret`、Git Token、镜像站凭据等直接填写的密钥值必须由后端加密为内部引用保存。生产环境必须配置稳定的 `SECRET_ENCRYPTION_KEY`，未配置时 API/worker 必须拒绝启动；开发环境必须显式 `APP_ENV=development` 才允许使用本地默认密钥。
- Git/OIDC OAuth state 必须走 Redis TTL 临时存储，回调时消费后删除；Redis 是平台强依赖，不做内存兜底，禁止新增 `git_oauth_states`、`oidc_auth_states` 这类临时握手业务表。
- 删除、回滚、生产发布等危险操作不使用普通表单直接提交，必须经过确认 Dialog。

## 5. 认证和权限

认证方式支持：

```text
Local Account
Casdoor OIDC/OAuth2
Generic OIDC/OAuth2
```

职责划分：

```text
Local Account: 平台内置账号，由管理员创建、邀请或导入
OIDC Provider: 身份认证、登录、多因子、外部组织和组信息
平台: 登录准入、项目权限、应用权限、发布权限、审计记录
```

内部平台默认不开放自由注册。

管理员初始化采用混合策略：

- 开发模式自动创建开发管理员，默认账号用于本地调试。
- 生产模式不自动创建开发账号。
- 生产模式不得返回、预填或展示开发默认账号、默认密码等调试提示。
- 前端不得硬编码开发默认账号；如需展示开发提示，必须使用后端在 development 模式下显式返回的字段。
- 生产模式如果没有任何 PlatformAdmin，前端进入初始化页面，调用一次性初始化接口创建首个管理员。
- 首个管理员创建成功后立即建立登录会话。
- 只要系统已经存在 PlatformAdmin，初始化接口必须拒绝再次调用。
- 显式 `APP_ENV` 优先决定运行模式；未配置时必须默认 production。开发模式必须显式设置 `APP_ENV=development`，禁止根据 `go run` 自动推断开发模式。

准入要求：

- 本地账号不支持公开注册，必须由 PlatformAdmin 创建、邀请或导入。
- OIDC 登录成功后必须执行平台准入校验。
- OIDC 必须支持允许组白名单。
- OIDC group claim 名称可配置，默认优先使用 `groups`。
- OIDC Provider 必须通过后台系统配置，不通过环境变量配置。
- 支持多个 OIDC Provider，并支持登录页选择 Provider。
- OIDC 外部身份必须落库为独立绑定关系，不直接把 User 固定为单一 `authType`。
- OIDC 首次登录绑定现有用户时，只允许使用非空且已验证邮箱匹配；空邮箱、未验证邮箱或多用户重复邮箱必须拒绝自动绑定。
- 登录态下支持用户主动绑定第三方登录，绑定流程必须校验 state、provider、external subject 和邮箱信息。
- 可选支持邮箱域白名单和邀请邮箱白名单。
- 未命中准入策略的用户不得自动创建平台账号。
- 准入失败必须记录 AuditLog。
- OIDC 组不能直接替代 ProjectMember 权限；可以作为默认授权或成员同步来源，但最终权限必须落入平台 RBAC。
- 运行模式必须可区分开发和生产。开发模式允许开发账号快捷登录；生产模式禁止开发账号快捷登录。运行模式通过显式 `APP_ENV` 实现，未设置时默认 production。

配置模型：

```text
AuthProvider
  - type: local | oidc
  - name
  - enabled
  - issuerUrl
  - clientId
  - clientSecret: write-only input
  - clientSecretSet
  - groupClaim
  - emailClaim
  - usernameClaim

AuthAdmissionPolicy
  - allowLocalLogin
  - allowOidcLogin
  - allowSelfRegistration: false
  - allowedEmailDomains
  - allowedOidcGroups
  - invitedEmails
  - defaultRole
```

权限模型建议：

| 角色 | 权限 |
| --- | --- |
| PlatformAdmin | 管理全局集成、集群、制品库、系统配置 |
| ProjectAdmin | 管理项目成员、应用、环境 |
| Developer | 查看应用、触发构建、发布非生产环境 |
| Operator | 部署、回滚、查看运行状态 |
| Viewer | 只读查看 |

MVP 项目成员角色：

| 项目角色 | 权限 |
| --- | --- |
| owner | 管理项目、删除项目、管理应用 |
| admin | 更新项目、管理应用 |
| developer | 管理应用 |
| viewer | 只读查看项目和应用 |

后端必须按项目角色执行权限校验，前端隐藏入口只作为体验优化。

生产发布建议单独控制：

- `prod.deploy`
- `prod.rollback`
- `secret.write`
- `cluster.manage`

## 6. 外部系统适配要求

### 6.1 代码库

第一阶段支持：

```text
Gitea + GitHub
```

后续支持：

```text
GitLab
```

抽象接口：

```text
CodeProvider
```

代码平台授权模型：

```text
GitProvider
  - type: github | gitea | gitlab
  - baseUrl
  - authType: oauth | github-app | pat
  - clientId
  - clientSecret: write-only input
  - clientSecretSet

GitAccount
  - userId
  - providerId
  - externalUserId
  - username
  - accessToken: write-only input
  - refreshToken: write-only input
  - accessTokenSet
  - refreshTokenSet
  - accessScope: personal | provider
  - scopes
  - expiresAt
```

要求：

- 支持跳转授权连接代码账户。
- 支持同一平台用户连接多个 Git 账户。
- RepositoryBinding 必须保存 `gitAccountId`。
- 可选 Actions Provider 的 workflow 写入、secrets 设置、workflow dispatch 必须使用绑定账户。
- token 不保存为业务表明文字段，只通过加密凭据引用。
- PAT 仅作为高级兜底方式，不作为普通主流程。

核心能力：

- 获取仓库列表
- 获取分支和提交
- 创建 Webhook
- 读取文件内容
- 写入或更新 CI workflow

### 6.2 构建与 CI

第一阶段主路径：

```text
Platform Builder Agent + executor container + BuildKit rootless
```

抽象接口：

```text
BuildProvider
```

核心能力：

- 接收 Git webhook
- 创建 BuildRun
- Builder Agent 领取 BuildJob
- 启动一次性 executor 容器
- 使用 BuildKit rootless 构建镜像
- 推送镜像到选择的镜像站
- 查询 BuildRun 状态
- 查询构建日志
- 取消构建
- 清理 executor 容器和临时 workspace
- 关联 commit 和镜像 tag

MVP 构建默认值：

```text
cpuRequest: 500m
cpuLimit: 1
memoryRequest: 1Gi
memoryLimit: 2Gi
timeout: 20m
retry: 0
successJobRetention: 10m
failedJobRetention: 24h
failureLogTail: 200
```

构建扩展要求：

- 默认值允许用户按权限修改。
- Project 需要预留构建资源配额。
- 资源按 CPU 核/月和内存 GiB/月计量。
- 内部货币单位使用 `credit`。
- MVP 不实现完整计费系统，但 BuildRun 必须记录资源消耗和 credits。
- MVP 不做持久化构建缓存，但 BuildProvider 接口必须预留 cache 配置。
- Build trigger 支持 `push`、`tag`、`manual`、`api`。
- API 触发使用平台 Access Token，不使用 JWT。
- Access Token 只保存 hash，支持 scope、过期时间和撤销。
- Access Token 管理归入账号页的“个人令牌”子 tab；全局侧边栏不单独放 Access Token 入口。
- Webhook 必须校验签名，并且只接受绑定仓库事件。

构建网络与 SSRF 防护要求：

- 构建 Job 默认必须使用 restricted 网络模式。
- 构建 Job 不允许默认拥有无限制 egress。
- 平台需要实现 BuildNetworkPolicy 配置模型。
- 平台需要实现 NetworkPolicyProvider 抽象，用于适配不同集群的 CNI、egress gateway 或代理方案。
- MVP 至少支持 CIDR + port 级别的出口控制。
- 域名白名单必须作为配置能力保留；如果当前 CNI 不支持域名策略，需要通过出口代理或后续 provider 实现。
- 默认允许公开 Git、公开 registry、公开包管理源和全局镜像加速源。
- 内网 registry 或内网包镜像源可以访问，但默认只允许私有网段 TCP 443。
- 私有网段非 443 端口默认禁止。
- 默认禁止 `169.254.169.254/32`、`127.0.0.0/8`、Kubernetes API Server 和 Service CIDR。
- DNS 访问必须限制到平台配置的可信 DNS 解析器。
- 被网络策略拒绝的访问需要记录审计事件。

配置模型：

```text
BuildNetworkPolicy
  - mode: restricted | open
  - allowedDomains: string[]
  - allowedCIDRs: string[]
  - deniedCIDRs: string[]
  - allowedPorts: number[]
  - allowPrivateCIDRHttps: boolean
```

Actions Provider 额外能力：

- 生成 workflow
- 写入或更新 CI workflow
- 触发 workflow dispatch
- 查询 workflow run/jobs/logs
- 生成 workflow run 和 job log 外部链接

### 6.3 制品库

第一阶段主路径：

```text
Harbor
```

后续可扩展：

```text
Gitea Registry
DockerHub
```

抽象接口：

```text
RegistryProvider
```

镜像站范围：

```text
global
project
user
```

要求：

- 支持全局默认镜像站。
- 支持项目级镜像站。
- 支持用户自定义镜像站。
- workflow 编辑时可以选择镜像推送目标。
- 已有镜像部署时可以选择镜像站或填写完整镜像地址。
- 镜像站凭据通过独立凭据对象管理，业务接口不回显 secret 引用字段。
- 镜像站选择优先级为应用默认、项目默认、全局默认。
- MVP 数据模型包括 `ArtifactRegistry`、`RegistryCredential` 和 `ContainerImage`。
- `RegistryCredential.password` / `token` 必须作为 write-only 输入，加密后保存为内部 secret 引用，接口只返回 `passwordSet` / `tokenSet`。
- Registry 测试使用 Docker Registry `/v2/` endpoint，必须设置请求超时；内网地址默认只允许 TCP 443，开发模式允许 localhost。

核心能力：

- 创建或绑定项目
- 查询镜像仓库
- 查询 tag、digest
- 查询漏洞扫描状态
- 管理 robot account 引用
- 测试 registry 凭据
- 生成 workflow 使用的登录参数
- 生成 Kubernetes imagePullSecret 引用

### 6.4 运行时和网关

第一阶段支持：

```text
Kubernetes/K3s + Ingress
```

抽象接口：

```text
RuntimeProvider
GatewayProvider
```

核心能力：

- 创建命名空间
- 应用 Deployment/Service/ConfigMap/Secret
- 应用 Ingress
- 查询 rollout 状态
- 查询 Pod 状态
- 回滚到指定镜像

域名和证书要求：

- 默认域名格式为 `{appSlug}-{projectSlug}.{stage}.{rootDomain}`。
- 用户自定义域名必须支持。
- 自定义域名通过 CNAME 指向平台生成的目标地址。
- HTTPS 证书优先使用 HTTP Challenge。
- MVP 不做 DNS Challenge。
- 用户可以选择 HTTP 访问，不申请证书。

## 7. 暂缓引入

| 技术 | 暂缓原因 |
| --- | --- |
| GraphQL | 权限和审计复杂度高，MVP 不需要 |
| Next.js | 控制台不需要 SSR，Vite SPA 更简单 |
| Redux | 服务端状态为主，TanStack Query 足够 |
| Temporal | 很强，但 MVP 任务复杂度还不够 |
| Kubernetes CRD | 领域模型未稳定前不宜上 CRD |
| Service Mesh | 第一阶段只需要 Ingress 暴露 |
| Vault | 可作为后续密钥增强，第一阶段先支持 Secret 引用 |
| Docker Compose Runtime | 第一阶段聚焦 Kubernetes/K3s 主路径 |

## 8. 原则应用说明

KISS：

- 后端使用 Go + Gin，不引入复杂全栈框架。
- 前端使用 Vite SPA，不引入 SSR。
- 页面模块就近组织组件、hooks、schema 和文案。

YAGNI：

- 第一阶段不做 CRD、Service Mesh、Temporal、插件系统。
- 不提前建设复杂设计系统，只从 shadcn/ui 按需添加组件。
- Docker Compose 运行时暂缓，避免主路径变复杂。

DRY：

- 通过 OpenAPI 统一前后端接口类型。
- 通过 Provider 接口统一外部系统适配。
- 跨页面组件真实复用后再上提到 `components/common`。

SOLID：

- 业务模块依赖 Provider 抽象，不依赖具体 Gitea、Harbor、Kubernetes 实现。
- 页面负责组合，基础组件负责通用 UI，API hooks 负责数据访问。
- i18n、toast、auth、api client 分别独立，避免页面组件承担过多职责。
