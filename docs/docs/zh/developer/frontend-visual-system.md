# 控制台视觉体系

Luna DevOps 在 shadcn/ui 和 Tailwind CSS 之上增加了一层页面级视觉原语。它们只负责内容层级、布局和视觉语义，不承载查询、权限或提交逻辑。

## Design Token

Tailwind v4 token 集中维护在 `web/src/styles/design-tokens.css`，品牌色阶仍由 `brand-themes.css` 管理。组件只消费语义 utility，不直接绑定具体色值或重复维护页面尺寸。

| 职责 | Tailwind utility / token | 使用范围 |
| --- | --- | --- |
| 表面与文本 | `surface-*`、`foreground`、`muted-*` | 页面画布、实体容器和弱层级内容 |
| 品牌交互 | `primary-*` | 主操作、链接、焦点和选中态 |
| 状态 | `success-*`、`warning-*`、`danger-*`、`info-*` | 不随品牌色变化的状态反馈 |
| 结构线 | `separator-strong/subtle` | 区块边界与同类重复项分隔 |
| 圆角 | `rounded-control/container/feature` | 控件、普通业务容器和重点容器 |
| 间距 | `gap-inline/field/group/section`、`p-group/section` | 行内、字段、相关区块和页面主区块 |
| 页面留白 | `px-page-inline`、`py-page-block` | token 自身响应移动、中屏和桌面断点 |
| 动效 | `duration-fast/standard/slow`、`ease-standard/emphasized` | 控件反馈、表面过渡与强调型位移 |
| 抬升 | `shadow-raised/overlay` | 交互抬升和浮层 |

公共布局和语义组件负责消费 token。业务页面优先组合 `PageShell`、`PageChrome`、`Surface`、`Section`、`MetricGroup`、`DataList` 与 `FormActions`，不要用 `@apply` 复制整组业务组件样式，也不要为单页新增另一套 token。shadcn 基础控件保留自身尺寸体系，页面级 token 不覆盖其可访问性和交互约束。

交互反馈默认使用 `duration-fast`，卡片背景、阴影和轻量位移使用 `duration-standard`，较大区域或分步转场才使用 `duration-slow`。常规颜色变化使用 `ease-standard`，带位移或缩放的强调反馈使用 `ease-emphasized`；不要在业务页面新增任意毫秒值。系统启用减少动态效果时，全局样式会把动画与过渡时长归零，位移和缩放应配合 `motion-safe`。

## 页面模板

新增页面应先归入以下类型，再选择根布局：

| 页面类型 | `PageShell` 宽度 | 推荐结构 |
| --- | --- | --- |
| 资源列表 | `full` | `PageChrome` + `DataList` |
| 看板与概览 | `content` | 待处理事项 + `MetricGroup` + `Section` |
| 设置 | `settings` | `ContentTabs` + `Section` / `Surface` |
| 日志、终端、拓扑 | `tool` | 工具栏 + 内嵌工作区 |

`DataList` 本身就是完整列表，不需要额外套一层 Card。公共 `TableFrame` 使用同一个边框和圆角容纳可滚动表体与可选表尾，分页位于表尾，不在表格外再生成独立容器。工具区、表头、正文与表尾各自维护内部留白，避免在表格四周形成重复空白。页面默认只保留一个实心主操作，搜索、筛选、排序和刷新应放入 `DataList.toolbar` 或 `ContentTabs.tools`。

## 表面层级

页面使用以下语义 token，不直接为业务容器选择临时背景色：

- `surface-base`：页面底色。
- `surface-raised`：主内容和浮起表面。
- `surface-subtle`：弱分组和 hover。
- `surface-inset`：代码、日志、诊断和筛选工作区。

业务区块优先使用 `Surface` 与 `Section`。`Card` 仍是基础组件，适合独立重复条目，但不应成为所有内容的默认外壳，也不要形成没有业务含义的 Card 嵌套。普通业务容器、指标组、状态提示和资源陈列卡片的外层统一不绘制边框，以实体表面和圆角建立层级；内部行分隔、左右分栏、表单控件及需要表达归属的局部结构仍保留语义边界。

浮层统一使用 `shadow-overlay`，需要与页面轻微分离的主表面使用 `shadow-raised`。

## 颜色职责

颜色分为三类：

1. Luna 品牌色用于 Logo 和固定品牌图形。
2. 交互主题色用于按钮、焦点、选中和 tab 指示器，可以跟随站点或用户偏好。
3. 成功、警告、危险和信息状态使用固定语义 token，不受个人主题色影响。

状态请复用 `StatusBadge`、`StatusValueBadge` 或 `Notice`。业务页面不要直接拼写 `red-*`、`amber-*`、`green-*` 等状态颜色；第三方品牌图标、终端和集中维护的图表色板除外。

控制台还提供“标准 / 简约”界面风格。标准模式将品牌边界色按低比例混入中性表面：浅色模式以接近白色的画布为主，深色模式以接近黑色的画布为主，并用递增混合比例区分导航悬停与选中态。简约模式把大面积画布、侧边栏和导航弱表面完全映射回中性白/灰表面，但不会改变主按钮、链接、焦点、Tab 指示器或状态色。界面风格由应用根节点统一解析并写入 `data-interface-style`，业务页面不得读取账号偏好后自行维护样式分支。

配色主题支持多色与单色两类稳定预设。多色主题通过 `primary`、`theme-secondary`、`theme-supporting` 和 `theme-highlight` 四个语义角色组织；布局层将后三个角色映射到选中表面和结构线，而 `workspace-background` 始终保持低饱和单色平面。业务组件只消费 `workspace-background`、`theme-selection-surface`、`separator-*` 等语义 token。单色主题将后三个角色回落到主色，因此不需要平行业务实现。选择器统一使用圆形色盘：多色主题由 SVG 精确路径绘制，主色占右下方 50% 半圆，半圆分界线按 45° 倾斜，其余三色均分另一半；主色底圆承接路径抗锯齿缝隙，禁止使用容易产生透明毛边的 `conic-gradient`。单色主题显示实心圆。亮色、暗色和跟随系统只控制明暗模式，集中放在账号个人设置中，不与配色主题组合成成倍增长的主题矩阵。

## 间距与密度

- 登录后主内容画布使用“横向宽松、纵向紧凑”的响应式页面内边距：移动端 `px-8 py-4`、中屏 `px-12 py-6`、桌面端 `px-16 py-8`；顶栏标题使用相同横向 padding。`PageShell` 只负责最大宽度和区块间距，不使用 `mx-auto` 或额外水平 padding，确保标题、看板、应用市场、列表和设置正文共享同一左侧基线。工具型页面如需占满剩余空间，应通过自身 viewport 结构处理，不覆盖全局页面留白。
- 桌面端页面标题放在内容工作区内，不使用独立全宽 topbar；标题与正文、Tab 和工具栏共享全局内容 padding 的左侧基线，并通过紧凑纵向间距形成同一组页面导航。移动端保留包含侧栏入口和标题的顶栏。
- 页面头部统一使用 `PageChrome`：第一行左侧标题、右侧页面工具，并保持稳定的最小高度，避免切换到没有工具的 Tab 时导航上下跳动；详情等层级页面可在标题下方传入一行“返回上级”导航，使用链接语义并由布局统一生成目标；只有传入 Tabs 时才单独显示末行，不为无 Tab 页面保留空白导航区域。`ContentTabs` 只承载 Tab 状态和内容切换，并把可选导航与工具交给 `PageChrome` 统一布局；中小屏工具回落到正文流中，返回导航仍保留在标题下方。
- `DataList` 使用与列表容器一致的实体表头，不使用整条灰色色块；表格内容轨道使用 `spacing-group` 向左右内收，不让内容碰到容器圆角边界。首个数据行和后续数据行都由自身顶边绘制 `separator-strong`，让表头到首行、数据行之间使用完全相同的元素、方向和 token；固定操作列边界与骨架线也使用同一 token。查询、筛选或批量操作工具区存在时，在工具区下方增加同样左右内收的结构线；只有标题时不绘制。末行下方不画线。行 hover 使用圆角语义表面，并让相邻分隔线同步融入 hover 表面；固定操作列表头继续继承同一表面。
- 侧栏品牌区使用 72px 高度，40px Logo 与品牌区顶部、左侧均保持 16px 留白，避免横纵边距失衡。
- 内容 Tab 的灰色基线与主题色高亮线共享同一底部坐标，两条线均使用圆角端点；Tab Trigger 不再额外绘制下边框，避免高亮线与基线产生悬空间隙。
- `SearchSelect` 与 `SearchMultiSelect` 通过 `size="default" | "sm"` 统一控制触发器、搜索栏、选项字号、纵向间距、图标和勾选框尺寸；密集筛选区使用 `sm`，普通表单保持 `default`，业务页面不要分别覆盖内部元素 class。
- `DataList` 不创建 Card 外壳；表格滚动视口和可选表尾由公共 `TableFrame` 使用单层原生语义边框与圆角裁剪共同持有。分页属于表尾，并固定在横向滚动区域之外，宽表格滚动时仍保持可见。禁止用嵌套背景与内边距模拟边框，避免 sticky 表头在圆角处产生抗锯齿接缝。表格内部仅保留数据行之间的必要结构线，表体与表尾之间只保留一条分隔线。列表没有标题、搜索、筛选或批量操作工具区时，`DataList` 自动在表格上方保留一个 `spacing-group`，业务页面不需要另加补偿间距。移动端表格单元格使用紧凑横向留白；单页列表只显示条目摘要，不展示无效的每页条数与翻页控件，多页列表再按响应式两层布局展示分页。
- 列表页的搜索、筛选、排序和刷新控件统一放入 `DataList` 表头工具区，并从左侧开始排列。页面标题已经表达列表上下文时，不要再显示“XX 列表”这类重复标题；只有同一页面存在多个需要区分的独立列表时才使用 `DataList.title`。创建等页面级主操作放在 `PageChrome` 标题右侧。不要在列表外再增加独立查询工具栏。`DataList.toolbar` 用于搜索框之外的筛选、排序和刷新组合。
- 主表单、设置面板和账号面板默认使用 `p-6`；目录卡片、指标卡片等高频扫描内容可以使用 `p-4` 或 `p-5`；`DataList`、日志、拓扑、终端和 iframe 外壳使用 `p-0`，由内部结构控制留白。不要在同一个容器同时叠加父级 padding 与子级补偿 margin。
- 页面主要区块使用 `gap-6`。
- 同一主题下的相关区块使用 `gap-4`。
- 表单字段和工具组合使用 `gap-3`。
- 行内按钮、Badge 和图标文字使用 `gap-2`。
- 紧凑元信息使用 `gap-1`。

优先使用 Tailwind 的标准间距与宽度 token，不为局部视觉微调引入任意像素值。

## 表单与操作区

- 设置类表单默认限制在 `max-w-3xl` 至 `max-w-4xl`，不要让短字段随内容区无限拉伸。
- 相关开关可以使用带轻量 `inset` 表面的选项组；字段较多时按业务含义分组，不用一张大 Card 等权包住所有内容。
- 页面表单的提交与取消操作统一使用 `FormActions`。桌面端按钮保持正常宽度并右对齐，移动端按钮才允许全宽。
- 同一设置页的不同 tab 必须把保存操作放在一致位置；默认位于当前表单末尾，不在部分 tab 使用顶栏、部分 tab 使用底部。
- 登录后的控制台统一由布局层为页面顶栏与整个内容画布提供 `workspace-background` 低饱和主题背景，页面组件不使用负间距或超大装饰容器模拟全屏。普通业务 `Surface` 与 `Card` 使用透明边框和实体表面形成扁平层级，不添加常驻阴影；阴影只用于弹窗、悬浮层、明确的 raised 表面和交互 hover。表格行、输入框、状态提示及需要明确归属的局部边界继续保留语义描边；标准模式下 `separator-*` 可由主题辅助色轻度染色，简约模式保持中性。
- 登录后应用最外层使用 `workspace-background` 单色主题背景，桌面侧边栏保持透明并直接继承全局画布，不额外绘制背景、右侧分隔边或菜单分组线；菜单类型通过分组标题与纵向留白区分。导航 hover 使用轻量实色 `sidebar-nav-hover`。单色主题和简约模式的选中态使用更深实色，多色主题在标准模式下使用主色到辅助色的低饱和渐变；渐变两端都必须保持完整选中表面，不使用透明或接近画布色的尾端，并始终保持“选中态强于 hover 态”的视觉权重。移动抽屉作为覆盖层仍使用实体主题背景。
- 界面风格采用“个人偏好 → 平台默认 → 标准模式”的解析顺序。账号保存空偏好表示持续跟随平台；显式选择标准或简约后覆盖平台默认。新增页面只使用现有表面与主题 token，不直接为简约模式增加条件 class。
- 侧边栏分组标题使用小于菜单项的弱层级字号、正常字重和较低对比度；不要让“工作台、资源、系统管理、个人”等分类名称与可点击菜单项争夺视觉注意力。
- 长表单的操作区使用顶部分隔线结束内容；Dialog 继续使用 `DialogFooter`，登录、注册等单任务窄屏流程可以保留全宽提交按钮。
- 表单按钮不得因为父级是 CSS Grid 而被默认拉伸成整行。

## 加载与空状态

加载状态按页面结构选择公共骨架：

- `AppLoadingState`：会话和公共配置初始化。
- `DataListSkeleton`：资源表头、数据行和分页轮廓。
- `OverviewSkeleton`：关注区、指标组和主次内容区。
- `SettingsSkeleton`：设置页 tab、标签和字段。
- `TemplateGridSkeleton`：应用市场工具栏与模板网格。
- `ToolViewportSkeleton`：iframe、日志、拓扑和终端工作区。

页面框架已经可用时只替换正文区域，禁止用一张大 Card 包裹单行“加载中”。`DataList` 在 `total === 0` 时会隐藏分页控件；首次配置场景使用可行动空状态，搜索或筛选无结果时使用紧凑空状态并提供清除条件入口。

## 操作与移动端

- 同一页面或 tab 默认只有一个实心主操作，其他同级操作使用 outline、ghost 或菜单。
- 看板和概览的异常必须在摘要层使用 `danger`、`warning`、`success` 或 `info` tone，并保留文字说明。
- 看板的大面积容器默认使用中性实体表面；语义色集中用于异常图标、关键数值和必要状态徽标，不应同时铺满提示区、指标格和状态标签。`Notice` 与 `MetricItem` 可通过 `variant="neutral"`、`surface="neutral"` 保留语义信息而降低综合色噪声。
- 看板按“风险摘要与活动指标 → 近期操作流与平台准备度 → 常用资源入口”组织阅读顺序。零值正常指标可以弱化，空活动区不得用固定高度制造大面积空白；常用资源在宽屏使用响应式网格，条目较多时通过明确的“查看全部”进入完整列表。
- 桌面筛选超过四项时，移动端使用 Sheet 或弹出层渐进披露；搜索、筛选入口和刷新保留在主页面。
- 高频移动端列表通过 `DataListColumn.mobile` 指定保留列，把次要信息放入主单元格第二行；复杂资源表格才保留横向滚动。
- 开发悬浮工具和其他 fixed/sticky 控件必须避让 Dialog、Sheet、toast、分页和固定操作列。

## 技术词汇

- 外部代码平台统一称为 `Git Provider`，身份认证平台统一称为 `OIDC Provider`，不在同类页面中交替使用“提供商”“平台”和裸 `Provider`。
- `Scope` 作为权限概念时展示为“权限范围”，需要帮助用户对应外部协议字段时写作“权限范围（Scopes）”。平台资源的可见边界继续称为“作用域”。
- 仓库字段优先展示“仓库所有者（Owner）”和“仓库名称（Repo）”；技术专名保留英文并在首次出现时提供中文语义。
- OIDC 的 Claim 字段使用“用户组字段（Group Claim）”“邮箱字段（Email Claim）”等中英组合，避免只显示英文缩写。
- 中文与英文、数字之间保留一个空格；API 枚举、命令、路径和变量名保持原始大小写。

## 验收

涉及公共视觉组件、页面模板或主题 token 的改动，至少检查：

- light 与 dark 模式。
- 默认主题色和一个非默认主题色。
- 1440px、1024px 和 390px 宽度。
- 空状态、异常状态、横向溢出和长文本。

视觉整改不能改变原有查询、权限、校验、提交和路由行为。
