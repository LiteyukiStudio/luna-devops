# Luna DevOps 前端视觉体系整改方案

本文将当前控制台的设计审美评审转化为可执行的工程方案。目标不是重做前端或替换 shadcn/ui，而是在现有设计 token、基础组件和业务页面之间补齐一层 Luna DevOps 产品级视觉规范，解决页面层级扁平、卡片使用泛化、间距与宽度不统一、品牌色边界模糊等问题。

本文属于实施前方案。执行过程中涉及用户可见行为、开发计划或验收状态变化时，应同步更新 `docs/` 和 `TODO.md`；仅调整本文不代表整改事项已经完成。

## 1. 当前结论

当前前端已经具备以下良好基础：

- shadcn/ui 基础组件、Tailwind CSS 和统一主题 token 已经接入。
- `DataList`、`ContentTabs`、状态 Badge、表单字段、空状态和错误状态已有公共实现。
- light、dark、system 三态主题、品牌色预设、移动端布局和减少动态效果均已覆盖。
- 管理列表的密度、固定操作列、分页和状态表达整体成熟。
- 登录页已有较完整的 Luna DevOps 品牌表达。

当前主要问题不是单个控件质量，而是业务页面自由组合基础组件后缺少统一的页面级规则：

1. `Card` 同时承担主容器、表格外壳、指标、提示、表单、画布和登录面板等不同职责，视觉权重趋同。
2. 页面根容器混用 `gap-4`、`gap-5`、`gap-6` 和 `space-y-4`，相似页面的节奏存在细微分裂。
3. 看板、项目概览和应用概览存在卡片嵌套及连续描边，异常信息与普通信息缺少明确优先级。
4. 设置页的内容列与页面级操作没有统一宽度边界，宽屏下构图重心分裂。
5. 品牌色、用户主题色和成功/警告/危险等状态色之间缺少明确边界。
6. 页面仍有零散的 `amber-*`、`red-*`、`green-*` 等状态样式；部分语义类缺少明确 token 定义。
7. 页面标题、区块标题、面板标题和列表工具栏仍存在多套临时布局。
8. 大型业务组件同时负责数据、交互和布局，使统一视觉规则的成本持续上升。

## 2. 整改目标与非目标

### 2.1 目标

- 建立页面背景、内容表面、内嵌工作区和浮层的清晰层级。
- 为列表页、概览页、设置页和工具工作区建立稳定模板。
- 让异常、风险和待处理事项比普通统计更容易被发现。
- 保持控制台克制、专业和高信息密度，同时增强 Luna DevOps 的产品辨识度。
- 复用现有 shadcn/ui 和公共组件，减少页面内重复的标题、工具栏、指标和提示结构。
- 保证品牌主题切换不会破坏状态语义和产品品牌识别。

### 2.2 非目标

- 不替换 React、Tailwind CSS、shadcn/ui 或现有路由体系。
- 不为了视觉整改改变后端 API、权限模型或业务数据结构。
- 不在第一阶段增加大规模插画、复杂动效或装饰性背景。
- 不把所有页面强行限制为相同宽度和相同密度。
- 不在一次改动中迁移全站页面。

## 3. 设计系统整改

### 3.1 补齐语义 token

在 `web/src/index.css` 和必要的主题样式中补齐以下语义。具体颜色应同时提供 light/dark 值，并继续由 Tailwind theme 暴露给组件。

表面层级：

```text
surface-base     页面底色
surface-raised   主内容、弹窗和需要强调的浮起表面
surface-subtle   列表 hover、次级分组和弱提示背景
surface-inset    日志、代码、筛选面板等内嵌工作区
```

状态语义：

```text
success / success-subtle / success-border
warning / warning-subtle / warning-border
info / info-subtle / info-border
danger / danger-subtle / danger-border
```

阴影只保留两个业务级别：

```text
shadow-raised    需要与页面背景分离的主表面
shadow-overlay   Dialog、Popover、Sheet 等浮层
```

实施检查：

- 检查 `bg-surface-subtle`、`text-success` 等现有类是否确实生成有效 CSS。
- 状态色不得由用户品牌主题覆盖。
- 终端、代码编辑器和 GitHub/Gitea/GitLab 等第三方品牌图标可以保留固定颜色。
- 图表颜色应优先从 CSS token 读取；确需固定色板时集中定义，不在渲染逻辑中散落十六进制颜色。

### 3.2 固定品牌色边界

将颜色职责拆成三层：

- **Luna 品牌色**：Logo、品牌插画、品牌识别图形，保持固定的 Luna 蓝/青体系。
- **交互主题色**：按钮、焦点、选中、标签指示器，可由站点默认值或个人偏好覆盖。
- **状态语义色**：成功、警告、危险、信息，始终表达固定语义，不随个人主题变化。

站点设置页需要明确展示“品牌识别色”和“控制台交互主题色”的区别。若产品仍允许管理员改变全站主题色，Logo 本身不应随主题色漂移。

### 3.3 统一间距语义

业务组件默认采用以下节奏：

| 场景 | 建议间距 |
| --- | --- |
| 页面主要区块 | `gap-6` |
| 同一主题下的相关区块 | `gap-4` |
| 表单字段、工具组合 | `gap-3` |
| 行内按钮、Badge、图标文字 | `gap-2` |
| 紧凑元信息 | `gap-1` |

页面不得为了视觉微调随意新增 `gap-5`、任意像素间距或相邻区域不一致的 padding。特殊工具工作区、画布和终端可以在组件内部定义专属节奏。

## 4. 新增公共视觉原语

建议在 `web/src/components/common` 中逐步新增以下组件。组件只负责布局和视觉语义，不包含具体业务查询。

### 4.1 `PageShell`

统一页面根布局和内容宽度。

```tsx
<PageShell width="full | content | settings | tool" spacing="normal | compact">
  {children}
</PageShell>
```

- `full`：资源表格和大型工作区，使用全部可用宽度。
- `content`：概览和普通详情，限制过宽文本与卡片扩张。
- `settings`：设置页，建议最大宽度约 1120px。
- `tool`：终端、拓扑、日志、iframe 等工具工作区。

### 4.2 `PageToolbar`

统一搜索、筛选、排序和页面主操作。移动端允许工具纵向排列，桌面端主操作位于最右侧。

规则：

- 同一页面默认最多一个实心主色操作。
- 搜索、范围、排序和刷新属于同一个工具栏。
- 高级筛选可折叠，不让大量等权输入框长期占据首屏。

### 4.3 `Surface`

替代业务页面直接把 `Card` 用作所有容器的做法。

```tsx
<Surface variant="plain | bordered | raised | inset">
```

- `plain`：通过留白和分隔线形成区块。
- `bordered`：表单或列表的主要边界。
- `raised`：真正需要强调的主内容。
- `inset`：内嵌工作区、弱背景和诊断内容。

`Card` 仍作为 shadcn/ui 基础组件保留；新业务页面优先表达表面语义，避免 `Card > Card` 嵌套。

### 4.4 `Section`

统一区块标题、说明、图标和工具操作。

```tsx
<Section title={...} description={...} icon={...} tools={...}>
  {children}
</Section>
```

用它替代页面中重复的 `flex items-center justify-between` 标题结构，并明确区分页面标题、区块标题和列表标题。

### 4.5 `MetricGroup` 与 `MetricItem`

统一看板、项目概览、应用概览和计费中的指标展示。

```tsx
<MetricItem
  label={...}
  value={...}
  meta={...}
  tone="neutral | success | warning | danger"
  href={...}
/>
```

规则：

- 异常指标允许获得更高视觉权重和明确行动入口。
- 零值指标应弱化，不和待处理异常占据完全相同的视觉权重。
- 统一数字字号、图标容器、状态位置和 hover 行为。

### 4.6 `Notice`

统一页面级提示，替换手写的 amber/red 提示框。

```tsx
<Notice tone="info | success | warning | danger" title={...} actions={...}>
  {description}
</Notice>
```

### 4.7 `SettingsShell` 与 `ToolShell`

- `SettingsShell` 统一设置页内容宽度、说明区和吸附操作区。
- `ToolShell` 统一日志、终端、拓扑、YAML 和运营 iframe 的工具栏与工作区边界。

## 5. 页面模板

### 5.1 资源列表页

适用页面：项目空间、事件、用户、身份源、代码库、镜像站、集群资源、Access Token。

```text
PageShell full
├── PageToolbar
│   ├── Search
│   ├── Filters / Sort
│   └── Primary action
└── DataList
```

要求：

- `DataList` 作为唯一列表外壳，不再无意义地额外套 `Card`。
- 操作列继续使用明确按钮或菜单，不把整行做成编辑入口。
- 工具栏和列表之间保持统一间距。
- 只有存在独立业务分区时才显示列表标题。

### 5.2 工作台与概览页

适用页面：平台看板、项目空间概览、应用概览、计费概览。

```text
PageShell content
├── AttentionZone
├── MetricGroup
├── PrimaryWorkspace
└── SecondarySections
```

要求：

- 待处理异常先于普通统计。
- 不连续使用多层描边卡片。
- 最近活动优先使用行列表，不为每一行创建独立卡片。
- 引导模块只在流程未完成时显示；完成后收敛为状态摘要或隐藏。

### 5.3 设置页

适用页面：站点设置、账号设置、注册、安全、身份源配置。

```text
PageShell settings
├── ContentTabs
└── SettingsShell
    ├── SettingsSection
    ├── Fields
    └── StickyActions
```

要求：

- 设置内容最大宽度和操作按钮使用同一右边界。
- 保存按钮不悬挂在远离表单的页面最右侧。
- 多个配置区优先通过 section 与分隔线组织，不为每段都增加卡片。
- 未修改、校验失败或保存中时继续遵守现有 disabled 规则。

### 5.4 工具工作区

适用页面：构建日志、运行终端、Web Console、拓扑、YAML、运营面板。

```text
PageShell tool
└── ToolShell
    ├── ToolToolbar
    └── ToolViewport
```

终端可以保持固定深色主题。工具栏、全屏、复制、下载、刷新等操作保持统一位置和尺寸。

## 7. 分阶段实施计划

### 阶段 A：基础 token 与公共组件

范围：

- 补齐表面、状态和阴影 token。
- 修复无效或未定义的语义样式类。
- 新增 `PageShell`、`PageToolbar`、`Surface`、`Section`、`MetricGroup` 和 `Notice`。
- 为公共组件补充用途、边界和示例注释。
- 在文档站增加控制台视觉与页面模板说明。

完成标准：

- light/dark 下全部语义 token 有效。
- 个人主题切换不改变状态含义。
- 公共组件无硬编码用户可见文本。
- 不改变业务流程。

### 阶段 B：三个样板页

按以下顺序实施：

1. 项目空间列表：验证列表页模板。
2. 平台看板：验证概览模板和指标优先级。
3. 全局设置：验证设置页宽度与操作区。

三个样板页验收后再复盘公共组件 API，避免直接铺开全站后反复返工。

### 阶段 C：同类页面迁移

列表批次：

- 事件、用户、身份源、代码库、镜像站、集群、Access Token。

概览批次：

- 项目空间概览、应用概览、计费概览。

工具批次：

- 构建日志、终端、拓扑、YAML、运营面板。

每个批次作为一个独立可验收目标推进，不在同一轮无边界修改全部页面。

### 阶段 D：品牌细化

- 建立 Luna DevOps 空状态插画或轻量品牌图形。
- 优化应用市场模板卡。
- 统一登录页与控制台的品牌衔接。
- 检查品牌 Logo、交互主题色和状态色边界。

品牌细化应在页面层级整改完成后进行，避免只给现有结构增加装饰。

## 8. 建议加入 `AGENTS.md` 的长期开发规范

以下规则建议在阶段 A 的方案评审通过后，提炼并加入项目级 `AGENTS.md`。加入时保持规则简短、可执行，不把本文整段复制过去。

### 8.1 页面模板

- 新增前端页面必须归入资源列表页、工作台/概览页、设置页或工具工作区之一，并优先复用对应公共布局组件。
- 页面根布局统一使用 `PageShell`；不得在页面中自由定义互相冲突的最大宽度和根间距。

### 8.2 表面与卡片

- 业务页面优先使用有语义的 `Surface`、`Section`、`MetricGroup` 等公共组件；shadcn `Card` 只作为基础组件，不作为所有内容的默认外壳。
- 禁止无明确语义的 `Card > Card` 嵌套；同一区域连续出现三个以上小卡片时，应优先评估指标组、行列表或分区布局。
- `DataList` 默认作为列表唯一外壳，禁止仅为增加边框再次套 Card。

### 8.3 颜色

- 成功、警告、危险和信息提示必须使用统一语义 token 或公共组件，页面不得直接拼写 `red-*`、`amber-*`、`green-*` 等状态样式；终端、图表固定色板和第三方品牌色例外。
- 用户主题色只影响交互状态，不得覆盖状态语义色和 Luna DevOps 固定品牌识别色。

### 8.4 页面操作与层级

- 同一页面默认最多一个实心主色主操作；搜索、筛选、排序、刷新和主操作应统一进入 `PageToolbar` 或 `ContentTabs.tools`。
- 看板和概览页必须优先展示异常及待处理事项；零值统计不得与异常指标保持完全相同的视觉权重。
- 设置页保存操作必须与设置内容列对齐，长设置页优先使用统一操作栏。

### 8.5 间距与宽度

- 页面主要区块、相关区块、表单字段和行内元素分别优先使用 `gap-6`、`gap-4`、`gap-3`、`gap-2`；新增任意间距前应确认现有语义间距无法满足。
- 列表页允许使用全宽；文本、设置和普通概览必须使用对应内容宽度模板，避免在超宽屏无限拉伸。

### 8.6 视觉验收

- 涉及公共视觉组件、页面模板或主题 token 的改动，必须至少验收 light/dark、桌面/移动端和一个非默认品牌主题。
- 大范围视觉改动必须对看板、项目列表、应用详情和设置页保留验收截图或等价的可追溯记录。

## 9. 验收方案

本整改预计修改文件数超过 8 个并跨多个前端业务域，实施阶段必须执行完整验证。

代码检查：

```bash
pnpm --dir web lint
pnpm --dir web build
```

浏览器尺寸：

- 1440px 桌面。
- 1024px 窄桌面。
- 390px 移动端。

主题组合：

- light。
- dark。
- system。
- 默认 Blue 交互主题。
- 至少一个暖色或粉色个人主题。

固定验收页面：

```text
/dashboard
/projects
/events
/projects/:projectId
/projects/:projectId/apps/:applicationId
/settings/site
/login
```

固定状态：

- 看板有异常与无异常。
- 列表空状态、少量数据、分页和横向溢出。
- 设置表单未修改、已修改、校验失败和保存中。
- 长表单 Dialog。
- 项目与应用 tabs 切换。
- 日志、终端和拓扑普通/全屏状态。
- 桌面侧边栏和移动端侧边抽屉。

## 10. 完成定义

视觉体系整改只有满足以下条件才可标记完成：

- 四类页面模板已经有公共实现和实际样板页。
- 主要业务页面不再依赖自由组合 Card 建立层级。
- 状态色、品牌色和交互主题色边界清晰。
- 看板和应用概览能够明显区分异常、普通状态和零值。
- 设置页内容与操作区在宽屏和移动端均保持一致重心。
- 相关用户可见变更已同步到 `docs/`，计划和验收状态已同步到 `TODO.md`。
- `pnpm --dir web lint` 与 `pnpm --dir web build` 无新增 error 或 warning。
- 固定页面完成多尺寸、多主题浏览器验收并形成可追溯记录。
